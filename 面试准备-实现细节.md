## 1. LangGraph 条件路由中的 State Mutation Bug（高频追问点）

**现象**：Critic 评分低于阈值时，`_supervisor_router` 返回 `"planner"`，但 `research_round` 没有递增，导致无限循环或提前终止。

**根因**：LangGraph 的 conditional edge 路由函数（`_supervisor_router`）是纯读函数——它在 **node 执行前** 被调用，只读取当前 state 决定去哪，**不参与状态写入**。如果在 router 中修改 state，框架会静默丢弃该修改。

**解决方案**：将 `research_round` 的递增从 router 函数移到 Critic node 的返回值中：

```python
# critic_node 返回时递增 round（正确做法）
return {
    "quality_score": overall,
    "research_round": current_round + 1,  # ← 在 node 返回值中修改，框架会 merge 到 state
    ...
}

# _supervisor_router 只做纯读取（正确做法）
def _supervisor_router(state):
    if score >= threshold or current_round > max_rounds:
        return "formatter"
    return "planner"
```

**反思点**：LangGraph 的 state 更新机制是「node 返回 dict → 框架按 reducer 合并」而非「直接 mutate」。理解 `TypedDict + reducer` 的状态管理模型是正确使用 LangGraph 的前提。

## 2. LLMProvider 多 Provider 桥接的核心难点

### 2.1 消息格式差异

不同 Provider 对同一语义的表达完全不同，这是桥接层的核心复杂度：

| 概念 | OpenAI 格式 | Anthropic 格式 |
|------|------------|---------------|
| 文本回复 | `{"role": "assistant", "content": "text"}` | `{"role": "assistant", "content": [{"type": "text", "text": "text"}]}` |
| 工具调用 | `tool_calls` 数组在主消息外 | `{"type": "tool_use", "id": ..., "name": ..., "input": {...}}` 混在 `content` 列表里 |
| 工具结果 | `{"role": "tool", "tool_call_id": "..."}` | `{"role": "user", "content": [{"type": "tool_result", "tool_use_id": "...", "content": "..."}]}` |

**Anthropic 没有 `tool` role**，工具结果必须包在 `user` 消息的 `tool_result` content block 中。这是双向转换中最容易出错的地方。

### 2.2 工具格式差异

```
OpenAI function-calling 格式:
{"type": "function", "function": {"name": "...", "description": "...", "parameters": {...}}}

Anthropic tool 格式:
{"name": "...", "description": "...", "input_schema": {...}}
```

### 2.3 Provider 分发策略

`LLMProvider.ask_tool()` 按 `self.provider` 分发到三个方法：
- `_ask_anthropic()` → `AsyncAnthropic.messages.create()`
- `_ask_openai()` → `AsyncOpenAI.chat.completions.create()`（也用于 DeepSeek）
- `_ask_ollama()` → `httpx.AsyncClient.post("/api/chat")`

每个方法独立构建请求参数、调用原生 SDK、解析响应、统一返回 `ToolCallResult`。

## 3. OpenManus Vendoring 中的 Pydantic 兼容问题

**问题**：OpenManus 的 `BaseAgent.model_validator` 会检查 `llm` 字段的类型——如果不是 `app.llm.LLM` 实例，就自动替换。

```python
# ❌ 正常传参会触发 model_validator，被替换为 OpenManus 自己的 LLM
agent = PlannerAgent(llm=my_provider, available_tools=tools)

# ✅ 绕过方案：Pydantic 初始化后再注入
agent = agent_cls(available_tools=tools)
agent.__dict__["llm"] = LLMProvider(llm_config)  # 绕过 validator
```

**为什么不用 `object.__setattr__`**？`__dict__` 直接操作更简单，且不需要 disable Pydantic 的 validate_assignment。

## 4. MCP 双模式工具架构

```
                    Agent 层
                       │
              ┌────────┴────────┐
              │  ask_tool()     │  ← 不感知工具部署方式
              └────────┬────────┘
                       │
         ┌─────────────┴─────────────┐
         │                           │
    In-Process 模式              MCP 模式
    (BaseTool.execute)      (stdio JSON-RPC 子进程)
         │                           │
         │                    ┌──────┴──────┐
         │                    │  MCPManager  │
         │                    │  • connect_stdio()
         │                    │  • 聚合 tools
         │                    │  • disconnect_all()
         │                    └──────┴──────┘
```

**设计要点**：
- `build_tool_collections()` 根据 `mcp.enabled` 配置决定走哪条路径
- In-Process：工具对象直接传给 `ToolCollection`，零网络开销
- MCP：工具通过 FastMCP 暴露为 stdio server，Agent 通过 `MCPClients.connect_stdio()` 连接
- 统一 MCP Server（`mcp_server.py`）把 9 个工具打包为单个进程，而不是每个工具一个进程（之前是 3 个 MCP Server，已合并为 1 个）
- **关键细节**：MCP 子进程不继承父进程环境变量，需要在 `mcp_server.py` 最开头 `load_dotenv()` 加载 `.env`

## 5. LLM JSON 输出的 3 层降级解析（`parse_review`）

LLM 经常不按规矩输出纯 JSON，`parse_review()` 设计了 3 层容错：

```python
# 第 1 层：正则提取 ```json ... ``` 代码块
json_match = re.search(r'```(?:json)?\s*(.*?)```', source, re.DOTALL)

# 第 2 层：直接搜索 JSON 对象（包含 overall_score 键）
json_match = re.search(r'\{[^{}]*"overall_score"[^{}]*\}', source, re.DOTALL)

# 第 3 层：简单正则提取 overall_score 数值（兜底）
score_match = re.search(r'(?:"overall_score"|overall_score)\s*[:=]\s*(\d+)', source)
```

每层失败后落入下一层，保证即使 LLM 输出格式严重偏离预期，系统也能推断出评分。

## 6. `Annotated[list, operator.add]` 的 Reducer 模式

```python
class ResearchState(TypedDict, total=False):
    search_results: Annotated[list[dict], operator.add]  # ← reducer 自动累加
    analyses: Annotated[list[dict], operator.add]
```

**为什么用 `operator.add` 而不是直接赋值？** LangGraph 的 state 合并规则：
- 普通字段：node 返回的 dict 覆盖 state 中的对应字段
- `Annotated[Type, reducer]`：node 返回的值通过 `reducer` 与原有值合并

`operator.add` 对 list 是 `list1 + list2`，所以每次 node 返回 `{"search_results": [新结果]}` 时，框架自动拼接到已有列表末尾。**如果不用 reducer，多次搜索的结果会互相覆盖，只保留最后一次。**

这正是并行搜索的基础——多个并行 Searcher 实例各自返回结果，`operator.add` reducer 自动合并为完整列表，无需手动拼接。

## 7. Searcher 并行搜索：asyncio.gather vs LangGraph Send API（方案选择重点）

### 7.1 问题

Searcher 节点需要对 N 个搜索查询逐一执行 Agent run，串行耗时 = N × 单查询耗时（~30s-2min/query）。

### 7.2 两种候选方案

**方案 A：LangGraph Send API（fan-out）**

```python
from langgraph.constants import Send

# planner → 条件路由返回 Send 列表，触发并行 search_single 节点
def fan_out(state):
    return [Send("search_single", {"query": q}) for q in state["search_queries"]]
```

需要新增 `search_single` 节点和 `aggregate` 合并节点，图拓扑从 7 个节点变为 9 个。

**方案 B：asyncio.gather + Semaphore（节点内并行，已实现）**

```python
semaphore = asyncio.Semaphore(max_parallel)

async def _search_one(query, index):
    async with semaphore:
        agent = searcher_factory()  # 每个查询独立 agent 实例
        await agent.run(query)
        return extract_result(agent)

results = await asyncio.gather(*[_search_one(q, i) for q in queries])
```

图拓扑不变，仅在 `searcher_node` 内部并行化。

### 7.3 选择 asyncio.gather 的理由

| 维度 | asyncio.gather | LangGraph Send API |
|------|---------------|-------------------|
| **图改动** | 0（节点内部） | +2 节点 + 1 条件路由 |
| **复杂度增量** | ~30 行 diff | ~80 行 diff + 调试链延长 |
| **agent 实例隔离** | 工厂函数创建 | Send 每次执行也需创建 |
| **状态合并** | `operator.add` reducer | 机制相同 |
| **单个查询超时** | `asyncio.timeout` 包裹 | LangGraph 节点级超时 |
| **可观测性** | 单节点日志，断点直观 | 多节点跳转，追踪链路长 |

**核心判断依据**：Searcher 的所有查询是**同质的**——同一套工具、同一 Agent 类型、同一执行逻辑。Send API 的价值在于**异构 fan-out**（例如 Planner 同时分发任务给 Searcher、Analyst、外部 API），每个目标节点有不同逻辑。同质任务用 Send API 是杀鸡用牛刀——多出来的节点纯粹是编排胶水，没有独立业务价值。

### 7.4 关键实现细节

**为什么每个查询要独立 agent 实例？** OpenManus 的 `ToolCallAgent` 内部维护了 `memory.messages` 列表（对话历史），以及 `state`（IDLE → RUNNING → FINISHED）和 `current_step`（步数计数器）。共享一个 agent 实例会导致：
- memory 污染（query1 的搜索结果混入 query2 的上下文）
- state 冲突（query1 将 agent 设为 FINISHED 后 query2 无法继续）
- 步数计数器跨越查询累积

**Semaphore 限流为什么用 3？** 大部分免费搜索 API（Brave、Tavily）有并发限制，3 是保守的默认值。通过 `max_parallel_searches` 配置，用户根据自己 API key 的限额调整。

**单个查询失败为什么不应该中断全部分搜索？** `asyncio.gather(return_exceptions=True)` 将异常打包为返回值，遍历结果时判断 `isinstance(r, Exception)`，失败查询返回错误标记 `{"query": ..., "content": "Error: ..."}` 而不是抛出。下游 Analyst 在分析时自行判断搜索质量。

## 8. asyncio.timeout + 进度轮询的并发模式

```python
async def _run_agent_with_progress(agent, request, agent_name, progress, timeout):
    _reset_agent(agent)

    async def _poll():
        while True:
            await asyncio.sleep(0.5)
            # 读取 agent.current_step / agent.state → 更新进度条

    async with asyncio.timeout(timeout):      # 墙钟超时保护
        poll_task = asyncio.create_task(_poll())  # 独立轮询协程
        try:
            result = await agent.run(request)     # 主任务
        finally:
            poll_task.cancel()                    # 主任务结束 → 停止轮询
```

**设计要点**：
- `asyncio.timeout` 装饰整个 `run()`，超时抛 `TimeoutError` → 设置 `agent.state = FINISHED` 避免悬挂
- 进度轮询是独立协程（`create_task`），500ms 间隔读取 agent 内部状态
- `finally` 中 `poll_task.cancel()` 保证轮询不会泄露
- 超时后 agent 状态标记为 FINISHED，防止 LangGraph 认为节点未完成而卡住

## 9. TypedDict(total=False) 的设计考量

```python
class ResearchState(TypedDict, total=False):
    topic: str
    search_results: Annotated[list[dict], operator.add]
    ...
```

**为什么 total=False？** LangGraph 的 `StateGraph.ainvoke(initial_state)` 要求初始 state 的字段可以不全。每个 node 返回的 dict 只包含它产出的字段（Planner 不返回 `quality_score`，Critic 不返回 `search_queries`），框架按字段名 merge。如果 `total=True`，初始化和 node 返回值都必须包含所有字段，非常繁琐。

## 10. 工具分发设计

`build_tool_collections()` 为每个 Agent 分配恰好需要的工具：

| Agent | 工具集 | 原因 |
|-------|-------|------|
| Planner | 只有 Terminate | 不需要外部工具 |
| Searcher | 7 个搜索工具 + Terminate | 唯一需要搜索的 Agent |
| Analyst | PythonExecute + Terminate | 可能需要沙箱计算 |
| Writer | CitationFormatter + Terminate | 需要引用格式化 |
| 其他 | 只有 Terminate | Agent 自己决定何时停止 |

工具最小化分配的好处：LLM 不会被无关工具干扰选择，减少误调用；减少 token 消耗（tools 定义占 input tokens）。

## 11. 面试话术建议

**当被问到"这个项目最大的技术挑战是什么"时：**

> 最大的挑战是三点：一是 Anthropic 和 OpenAI 的消息格式差异，tool_use/tool_result 用 content blocks 表达 vs tool_calls 数组，双向转换很多边界情况；二是 LangGraph conditional edge 中的 state mutation 被静默丢弃，当时花了较长时间定位，最终理解了它的 state reducer 模型；三是 MCP 子进程环境问题，子进程不继承父进程的 `.env`，需要在 mcp_server 入口先 `load_dotenv()`。

**当被问到"为什么不用 LangChain"时：**

> LangChain 为 OpenAI 格式设计，对 Anthropic 的 tool_use 支持需要 additional_kwargs 迂回传递，不够直观。而且引入了 5 个依赖包及其传递依赖（约 70 个包）。直接调用原生 SDK 更精准、更容易调试、依赖更少。

**当被问到"Searcher 并行搜索为什么用 asyncio.gather 而不是 LangGraph Send API"时：**

> Send API 更适合异构分发（不同节点不同逻辑），本项目 Searcher 的多个查询是同质的——同一套工具、同一 Agent 类型。用 Send API 需要新增 2 个节点 + 修改图拓扑，增加的只是编排胶水代码，没有独立业务价值。asyncio.gather 在节点内并行，改动量小（~30 行），且每个查询独立 agent 实例保证状态隔离。如果未来需要 Planner 同时向 Searcher 和外部 API 分发不同任务，再升级为 Send API。

**当被问到"如果要扩展这个系统，你会怎么做"时：**

> 一是 Searcher 节点内已实现 asyncio.gather 并行搜索，如果需要异构分发（不同节点并行执行不同逻辑），可以升级为 LangGraph Send API；二是接入外部 MCP Server（GitHub、Filesystem、PostgreSQL 等）扩展工具能力；三是增加 Human-in-the-loop 节点（LangGraph interrupt），让用户在关键决策点介入。
