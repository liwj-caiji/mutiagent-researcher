# Multi-Agent Research Assistant — 简历项目总结

## 项目简介

基于 **LangGraph** 编排的多智能体研究助手。用户输入任意研究主题，6 个专业化 LLM Agent 自动协作完成「主题分解 → 多源搜索 → 深度分析 → 综合归纳 → 报告撰写 → 质量审查」全流程，最终输出结构化 Markdown 研究报告。支持质量闭环（Critic 评分不通过自动迭代重研），每个 Agent 可独立配置不同的大语言模型。

- **代码规模**：~3,550 行 Python，34 个源文件，15 个直接依赖（安装后 73 个包）
- **技术栈**：LangGraph / Python / asyncio / Pydantic / MCP / Rich / Typer
- **LLM 支持**：Anthropic Claude / OpenAI GPT / DeepSeek / Ollama（通过原生 SDK 统一接入）

## 个人贡献

### 1. LangGraph 工作流编排（核心技术亮点）

- 基于 **StateGraph** 构建 Supervisor 模式的 6 节点流水线（Planner → Searcher → Analyst → Synthesizer → Writer → Critic → Formatter），使用 **TypedDict** 定义共享状态 `ResearchState`
- 实现 **条件路由**（`add_conditional_edges`）：Critic 评分低于阈值 → 自动回退到 Planner 执行下一轮研究，达成质量闭环；评分达标 → Formatter 输出最终报告
- 使用 `Annotated[list, operator.add]` 实现 `search_results` 和 `analyses` 字段的 **reducer 自动累加**，为并行搜索预留扩展能力
- 支持 **SQLite Checkpointer** 持久化状态，支持流水线中断后从断点恢复

### 2. Agent 运行时搭建

- **Vendoring OpenManus**：将外部 Agent 框架核心代码（~950 行）内联到项目，移除 121 个非必要传递依赖（browser-use、playwright、boto3 等重型包），将依赖数从 264 降至 143
- 编写 **LLMProvider 多 Provider 调用层**：封装各 Provider 原生 SDK（Anthropic/OpenAI），对外暴露统一的 `ask_tool()` 接口，实现 OpenManus 消息格式到各 Provider 原生格式的双向转换（重点是 Anthropic 独有的 tool_use/tool_result content blocks）
- 实现 **LLM 配置工厂**：通过 `AgentLLMConfig` 支持 4 种 Provider（Anthropic/OpenAI/DeepSeek/Ollama），每个 Agent 可独立配置模型、temperature、max_tokens。LLMProvider 直接调用各 Provider 原生 SDK（零中间层），内部按 Provider 分发到 `_ask_anthropic` / `_ask_openai` / `_ask_ollama`，各自处理消息格式双向转换

### 3. 工具层 + MCP 协议集成

- 实现 **9 个研究工具**：Brave Search、Tavily、DuckDuckGo、arXiv、Wikipedia、Jina Reader、Web Scraper、Python 沙箱执行、引文格式化，均继承统一 `BaseTool` 基类
- 设计 **双模式工具架构**：In-Process 模式（工具直接 Python 调用，零延迟，适合开发）和 MCP 模式（工具运行在独立子进程，通过 stdio JSON-RPC 通信，适合生产隔离），通过配置文件一键切换
- 使用 FastMCP 实现 **统一 MCP Server**，将 9 个工具通过 stdio 暴露，支持接入外部 MCP Server（如 GitHub、Filesystem、Puppeteer）
- 实现 **MCPManager** 管理子进程生命周期：启动连接、工具发现聚合、断开清理

### 4. 工程能力体现

- 调试并修复了多个运行时 bug：**LangGraph conditional edge 中 state mutation 被静默丢弃**（将 `research_round` 递增从 router 函数移到 node 返回值）、LLM 输出 JSON 解析脆弱（设计 3 层降级提取策略）、MCP 子进程无法加载 `.env` 导致 API key 缺失、Writer Agent 工具分配错误导致写入失败
- 清理 **3 处死代码**（`ToolRegistry`、`ReportSaverTool`、重复异常处理逻辑），修复异常被静默吞噬的问题
- 使用 **Rich Live** 实现实时进度面板，展示 6 个 Agent 的执行状态、步数进度和耗时，采用 **asyncio.timeout** 为每个 Agent 设置墙钟超时保护

## 面试可能追问的准备

**Q: LangGraph 和普通 DAG 的区别？**
A: LangGraph 天然支持条件路由和循环（DAG 无环），可以建模"研究 → 审查 → 不通过则重新研究"的迭代过程。此外提供了 checkpointing 状态持久化，支持从任意节点恢复执行。

**Q: 为什么用 Supervisor 模式而不是完全自治的多 Agent？**
A: 研究任务天然是线性的（先搜索再分析再写），Supervisor 模式简单可靠。如果后续需要并行搜索多个源，可以在 Searcher 节点内用 LangGraph 的 Send API 实现 fan-out，整体架构不受影响。

**Q: MCP 协议的价值是什么？**
A: MCP 标准化了 LLM 与外部工具的通信协议。工具运行在独立进程中，可以独立部署、升级、用不同语言编写。Agent 通过统一接口调用，不感知工具的具体位置和实现。

**Q: 为什么不用 LangChain？**
A: LangChain 主要为 OpenAI 格式设计，对 Anthropic 的 tool_use/tool_result content blocks 支持不完善（需要 additional_kwargs 迂回传递），且引入大量传递依赖（langchain-core、langchain-openai、langchain-anthropic 等 5 个包）。LLMProvider 直接调用各 Provider 原生 SDK（anthropic.AsyncAnthropic / openai.AsyncOpenAI / httpx），代码更精准、依赖更少、调试更方便。移除 LangChain 后减少了约 70 个传递依赖。

**Q: LLMProvider 的核心设计？**
A: Agent 框架期望 `ask_tool()` 返回 OpenAI 格式。但 Anthropic 的 tool_use/tool_result 用 content blocks 表达，和 OpenAI 的 tool_calls 数组完全不同。LLMProvider 内部按 Provider 分发到 `_ask_anthropic` / `_ask_openai` / `_ask_ollama`，各自处理消息和工具的格式双向转换，向上统一输出 ToolCallResult。支持 Anthropic/OpenAI/DeepSeek/Ollama 四种 Provider 任意切换。
